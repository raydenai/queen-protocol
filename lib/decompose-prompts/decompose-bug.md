# Decompose Bug — Meta-Prompt Template (v2.20.0)

> **Role:** Super-Queen auto-decomposer  
> **Input:** A user bug report (free text, e.g. "fix the dashboard chart that shows wrong totals")  
> **Output:** A shard graph in JSON matching §4 schema. Bugs are **single-shard by default**.

---

## 1. Input ingestion

Read the bug description. Extract:

1. **Symptom:** what is wrong? (wrong totals, 500 error, stale data, UI glitch)
2. **Location hint:** file, route, component, or subsystem mentioned.
3. **Repro steps:** if provided, note them; if absent, flag as assumption.
4. **Severity:** user-stated or inferred (data-loss → critical; cosmetic → low).

If the description is so vague that no file tree can be inferred (e.g. "something feels slow"), emit a **diagnostic shard** (`kind: diagnostic`) instead of a fix shard, and stop.

---

## 2. Default: single-shard bug fix

Bugs are **single-shard by default**. The default output is one shard:

```json
{
  "id": "s01",
  "title": "Fix: <symptom>",
  "kind": "code",
  "tags": ["bugfix", "<subsystem>"],
  "priority": "normal",
  "complexity": "obvious | mechanical | complex",
  "files_allowed": ["<inferred-file-or-dir>/**"],
  "depends_on": [],
  "estimated_lines": 20,
  "skills_required": [],
  "gates": ["pytest <relevant-scope>", "typecheck"],
  "deadline_minutes": 15,
  "routing_tier": "solo | review",
  "backend": "agent:codex-rescue | kimi-isolated | claude-ant"
}
```

**Why single-shard?** Bug fixes are read-coherence-bounded: the ant must understand the root cause, which usually requires reading multiple files to trace the fault. Splitting the read across ants fragments the causal chain and produces conflicting diagnoses.

---

## 3. When to expand beyond one shard

Expand **only** if the root cause spans subsystems. Trigger signals:

| Signal | Expansion |
|---|---|
| Bug is in `apps/api/` but symptom shows in `apps/dashboard/` AND the fix requires coordinated changes in both | 2 shards: backend-fix + frontend-fix, with `depends_on` backend → frontend |
| Bug involves a DB migration to correct bad data + code change to prevent recurrence | 2 shards: migration-senior (critical) + code-fix (depends on migration) |
| Bug is a race condition or distributed-system issue requiring instrumentation + fix + test | 1 shard still; add `diagnostic` sub-task as `kind: diagnostic` within same shard group, not separate write-shard |
| Bug fix requires renaming a symbol across 50+ files | 1 shard, tag `mechanical`, route `kimi-isolated` — still single-shard because `files_allowed` is a single glob |

**Anti-expansion rule:** Do NOT split a bug into "investigate" + "fix" shards. Investigation and fix belong to the same ant so the fixer has the full causal context. If the ant cannot diagnose within its deadline, mark `FAILED` and re-dispatch with broader `files_allowed` and a diagnostic prompt — do not pre-split.

---

## 4. Routing tier for bugs

Apply v2.18.0 thresholds with bug-specific adjustments:

| Condition | Tier | Backend |
|---|---|---|
| Stack trace provided, single file, <30 LOC | Solo | `agent:codex-rescue` (§1.1 signature) |
| No stack trace, location unclear | Diagnostic | `agent:codex-rescue` (foreground read-heavy) |
| 1–2 files, >30 LOC | Review | `claude-ant` or `agent:codex-rescue` + dual review at converge |
| 3+ files OR touches auth/payment/security | Race | Tournament (3-way) because correctness matters more than cost |
| Mechanical rename/format after root-cause fix | Solo | `kimi-isolated` if separated; usually absorbed into main fix shard |

---

## 5. Lane assignment for bugs

```text
IF stack trace present OR kind == diagnostic
   → agent:codex-rescue  (GPT-5.5 excels at trace reading)

IF tags & {security-critical, auth, payment}
   → tournament  (3-way race for triangulation)

IF complexity == mechanical AND root cause is known
   → kimi-isolated

DEFAULT
   → claude-ant  (full reasoning for ambiguous bugs)
```

```json
{
  "schema_version": "2.20.0",
  "colony_id": "<auto-generated: YYYY-MM-DD-bug-name>",
  "goal": "Fix: <symptom>",
  "decomposition_type": "bug",
  "status": "PLAN",
  "shards": [
    {
      "id": "s01",
      "title": "Fix: ...",
      "kind": "code",
      "tags": ["bugfix", ...],
      "priority": "normal | critical",
      "complexity": "obvious | mechanical | complex",
      "files_allowed": ["..."],
      "depends_on": [],
      "estimated_lines": 20,
      "skills_required": [],
      "gates": ["..."],
      "deadline_minutes": 15,
      "routing_tier": "solo | review | race",
      "backend": "..."
    }
  ],
  "dag_validation": {
    "acyclic": true,
    "all_deps_resolve": true,
    "files_overlap": false,
    "has_starting_shard": true
  },
  "assumptions": ["..."]

```

**If expanded to 2+ shards**, include the `dag_validation` block and ensure `depends_on` edges reflect the causal ordering (backend fix before frontend fix, migration before code).

---

## 7. Anti-patterns

1. **Splitting investigation from fix** → single shard wins. The fixer needs the investigation context.
2. **Routing a typo/one-liner through super-queen ceremony** → if <5 LOC and obvious, mark `queen-direct` and skip colony dispatch entirely.
3. **Expanding a CSS glitch into 3 shards** → unless it spans design-system + page + component, keep it single-shard.
4. **Forgetting a regression test gate** → every bug-fix shard must include a gate that proves the bug is fixed (e.g., `pytest -k test_dashboard_totals_correct`).
