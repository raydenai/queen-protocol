## Validation Rule Reference Table

| # | Rule | Severity | Reference | Implementation |
|---|---|---|---|---|
| 1 | **JSON schema** — every shard must have `id`, `tags`, `kind`, `priority`, `complexity`, `files_allowed`, `depends_on` | ERROR | §4 shard schema | Python schema pass over `shards[]` |
| 2 | **Type constraints** — `kind ∈ {code,review,diagnostic}`, `priority ∈ {critical,normal,low}`, `complexity ∈ {obvious,mechanical,complex}` | ERROR | §4 type contract | Enum membership checks |
| 3 | **Id uniqueness** — no duplicate `id` values within colony | ERROR | §2.2 | Set deduplication |
| 4 | **Anti-decomposition** — no two shards may have overlapping `files_allowed` globs/paths | ERROR | §30.3 HARD rule | Pairwise `_glob_overlap()` (exact match + prefix overlap + glob-prefix) |
| 5 | **Dependency resolution** — every `depends_on` entry must resolve to an existing shard `id` | ERROR | §2.2 DAG rule 1 | Lookup against shard-id set |
| 6 | **Dependency acyclicity** — `depends_on` graph must be a DAG | ERROR | §2.2 DAG rule 2 | DFS cycle detection (Tarjan-lite) |
| 7 | **Senior priority** — `priority: critical` shards cannot depend on non-critical shards | ERROR | §2.2 DAG rule 5 | Post-resolution priority comparison |
| 8 | **Tier validity** — shard size must map cleanly to v2.18.0 dispatch tier:<br>`<30 LOC + 1 file` → solo;<br>`1 file ≥30 LOC` OR `2 files` → review;<br>`3+ files` → race | WARN* | §30.4 lever 1 | `estimated_lines` + `len(files_allowed)`; skipped if `estimated_lines` absent |
| 9 | **Tier/backend alignment** — solo-sized shard should not carry a race backend; race-sized shard should not carry a solo backend | WARN | §4.2 routing | Heuristic substring match on `backend` |
| 10 | **Cap-awareness sanity** — `≥5` shards routed to the same paid backend triggers cap-exhaustion warning | WARN | §30.4 lever 8 | Frequency count on `backend` field |

\* Tier validity produces `WARN` when `estimated_lines` is missing; `ERROR` when the field is present but structurally nonsensical (negative, non-integer, or zero `files_allowed`).

## `--fix` Mode (Opt-In)

When `--fix` is passed and anti-decomposition violations exist, the tool:

1. Builds a union-find graph of shards connected by `files_allowed` overlap.
2. Merges each connected component into a single synthetic shard:
   - `id` → `idA+idB+...`
   - `files_allowed` → sorted union
   - `tags` → sorted union
   - `priority` → max(critical > normal > low)
   - `complexity` → max(complex > mechanical > obvious)
   - `kind` → preferential: code > review > diagnostic
   - `depends_on` → external deps only (edges inside the merged group are removed)
   - `backend` → most capable heuristic (tournament > branching > specialist > claude-ant > ...)
   - `estimated_lines` → sum of members
3. Re-runs the full validation suite on the merged graph.
4. Emits a `FIX:` annotation in stderr, then continues with normal output.

**Limitations (draft):**
- `--fix` does NOT re-write the source JSON file; it validates the in-memory fixed graph.
- The operator must manually port the merged graph back into `plan.json`.
- Synthetic shard IDs use `+` concatenation; downstream tools that don't expect `+` in IDs may need adjustment.

## Acceptance Criteria (Test Corpus)

| Fixture | Expected | Verdict | Expected stderr (non-quiet) |
|---|---|---|---|
| `test-01-solo-pass` | **PASS** | Exit 0 | `VALID: shard graph passes all checks.` |
| `test-02-dag-pass` | **PASS** | Exit 0 | `VALID: shard graph passes all checks.` (no warnings) |
| `test-03-cap-warning` | **PASS** | Exit 0 | `WARN: CAP-RISK: 6 shards routed to backend 'kimi-isolated' …` followed by `VALID: …` |
| `test-04-overlap-fail` | **FAIL** | Exit 1 | `VIOLATION: ANTI-DECOMPOSITION (§30.3): shards 's01' and 's02' …` |
| `test-05-cycle-fail` | **FAIL** | Exit 1 | `VIOLATION: DEP-CYCLE: s01 -> s02 -> s03 -> s01` |

### `--quiet` contract
- **Valid graphs:** absolute silence on stdout/stderr, exit 0.
- **Invalid graphs:** single line to stderr: `Shard graph invalid: N violation(s). Run without --quiet for details.`, exit 1.
- **Warnings only (e.g. cap-risk):** warnings are suppressed in `--quiet`; exit 0.

### `--fix` contract on `test-04-overlap-fail`
- Exit 0 after merge.
- Stderr contains: `FIX: merged overlapping shards into sub-shard groups. Re-validate passes.`
- The merged shard `id` becomes `s01+s02`.

## Backout Plan (False-Positive Mitigation)

If the validator blocks real colonies with false positives:

1. **Immediate bypass** — run with `--fix` to auto-merge overlap clusters; if the merge is acceptable, land manually.
2. **Per-colony override** — add `"validate_shard_graph": false` to `plan.json` root (not yet in spec; proposal for v2.20.2 if needed).
3. **Whitelist file** — introduce `~/.claude/state/colony/validate-whitelist.json` mapping `colony_id` → `[rule_id]` to skip specific rules per known-good colony.
4. **Rule rollback priority:**
   - Most likely false-positive: `_glob_overlap()` heuristic over-matches glob prefixes (e.g. `src/**/*.py` vs `src/tests/**/*.py`). **First fix:** refine prefix logic.
   - Second most likely: tier/backend alignment heuristic is too strict for specialist routing. **First fix:** demote tier-mismatch from WARN to INFO.
   - Third: cap-risk threshold of 5 is arbitrary per-lane. **First fix:** make threshold configurable via env var `QP_CAP_RISK_THRESHOLD`.
