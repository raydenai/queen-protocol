Two grounded analyses (Claude direct read of spec-kit templates + Kimi cross-comparison against QP §1-§29) converged on the same shortlist of high-leverage items. Most are evidence-shaped against existing QP failure modes in §28 and §29.6. Document parked rather than shipped because:

- §26.4 anti-pattern says: don't pile onto the protocol without direct QP incident evidence
- v2.15.x already added 4 patch bumps in 3 days — version-bump fatigue is real
- Each borrow has a specific trigger condition; ship only when the matching incident appears

### Key mechanism findings

**Constitution Check is lighter than initially framed.** From `plan-template.md`:

```
## Constitution Check
*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*
[Gates determined based on constitution file]
```

Plus a violations-justification table at the end. **It's documentation discipline, not a runtime gate.** That weakens the "must borrow this" framing — QP already has CLAUDE.md as source-of-truth #1.

**The genuinely novel patterns are:**

1. **Per-field `NEEDS CLARIFICATION` markers** in plan Technical Context — allows partial fills with explicit flags for follow-up
2. **Numbered Functional Requirements (FR-001+) + Success Criteria (SC-001+) + Edge Cases** in spec.md — structures intent in a way QP's `goal` string field doesn't
3. **`[Story]` traceability labels** on tasks — maps every task back to a numbered user story
4. **`/speckit.analyze` cross-artifact consistency check** — looks for coverage gaps between spec → plan → tasks

## Recommended phased ship (when triggers fire)

### Phase 1 (highest leverage — ship when trigger fires)

**Trigger:** next colony where an ant returns gate `status: PASS` but converge re-run catches gate lie (§29.6 pattern) → traceable to under-specified acceptance criteria.

**Ship:**

- New §27 (or appropriate §): **"Structured spec input (opt-in)"** — when colony tagged with `spec_required: true`, queen writes `spec.md` at PLAN with:
  - User stories prioritized (P1/P2/P3)
  - Per-story Acceptance Scenarios in Given/When/Then form
  - Numbered Functional Requirements (`FR-001`, `FR-002`, ...)
  - Numbered Success Criteria (`SC-001`, ...) — measurable, not "works correctly"
  - Edge Cases enumerated
  - Independent Test narrative
- New `validate-spec.py` script that checks numbered requirements are referenced in the ant prompts AND in the verify-gate criteria
- Auto-promote production-path tags (`migration|payment|auth|security-critical`) to `spec_required: true`

**Cost estimate:** ~2-3h work. ~80 lines doc, ~50 lines validator.

### Phase 2 (ship when MANIFEST→plan divergence incident occurs)

**Trigger:** next colony where MANIFEST.md and plan.json disagree on shard count, shard IDs, or scope (§28.2 pattern).

**Ship:**

- New `scripts/spec-plan-coverage.py` (the `/speckit.analyze` equivalent):
  - Reads `spec.md` (FR-001+, SC-001+, Edge Cases)
  - Reads `plan.json` (shards + gates)
  - Reports per-FR/SC/Edge: which shard addresses it; flag unaddressed items
  - Exit non-zero before DISPATCH if any production-path requirement is unaddressed

**Cost estimate:** ~3-4h work. ~150 lines.

### Phase 3 (ship when stack-fit mismatch incident occurs)

**Trigger:** next colony where ant uses wrong framework/lib (e.g., Codex assumes vitest but repo uses native strip-types — already documented in §29.6).

**Ship:**

- §10 plan.json schema enrichment: add `technical_context` object
  - Fields: `language_version`, `primary_dependencies`, `storage`, `testing`, `target_platform`, `project_type`, `performance_goals`, `constraints`, `scale_scope`
  - Each accepts a value OR `"NEEDS_CLARIFICATION: <note>"`
  - For `priority: critical` colonies, `NEEDS_CLARIFICATION` → exit non-zero before DISPATCH
  - For other colonies, WARN (consistent with §1.3 throughput posture)

**Cost estimate:** ~1h work. ~30 lines schema + ~20 lines validator.

### Phase 4+ (deprioritized — ship only on direct evidence)

- Constitution Check (light) — opt-in plan.json field
- Numbered checklist template for verify gates
- `[Story]` task traceability
- Repo-local `.specify` directory convention
- Constitution artifact template

## Anti-borrows (deliberately rejected)

These would actively hurt QP if adopted:

| Spec-kit item | Why NOT |
|---|---|
| `/speckit.taskstoissues` | QP's natural land is a PR, not a GitHub issue. Doesn't compose. |
| Phase numbering (Phase -1, Phase 0, Phase 1) | Fights QP's existing state machine (§2). Don't blend. |
| `specify` CLI as runtime | QP has its own state harness; replacing it is v3 rewrite, not calibration. |
| Single-agent integration model | QP's polymorphic worker pool (§4) is strictly more capable. Don't regress. |
| Complexity Tracking table (mandatory) | No observed QP failure traced to missing complexity justification. Add only if pattern emerges. |

## What WAS confirmed about spec-kit's actual leverage

Spec-kit's discipline is largely **template-driven, not runtime-enforced.** The "Constitution Check" is a placeholder section in plan-template.md, not a gate that runs code. The leverage comes from:

1. **Forcing the planner to write down structured intent** before HOW is discussed (spec.md before plan.md)
2. **Numbered requirements (FR/SC)** create traceability between intent and verification
3. **Per-field `NEEDS_CLARIFICATION` markers** are non-binary — ship the plan with some fields flagged, resolve them later, instead of blocking on first unknown

QP already enforces the strong-gates side runtime (verify-done, cross-shard audit, dispatch lock). Spec-kit complements with the upstream artifact-discipline side QP currently leaves to operator habit.

## When to revisit this proposal

**Re-read this file when any of these happen:**

- A QP colony ships PR with an ant gate-lie that an FR/SC numbering scheme would have caught → ship Phase 1
- A QP colony's MANIFEST and plan.json diverge on scope → ship Phase 2
- A QP colony picks the wrong stack lib because Technical Context wasn't captured → ship Phase 3
- Operator burns >1 hour debugging a colony failure that an upstream spec discipline would have caught → revisit prioritization

**Do NOT ship pre-emptively.** Even with the high-leverage scores above, the §26.4 anti-pattern wins: every protocol addition adds maintenance surface; evidence-shaped trigger conditions are the discipline.
