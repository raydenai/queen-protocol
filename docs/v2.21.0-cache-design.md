# Queen Protocol v2.21.0 — Cross-queen shared read cache (Design Draft)

> **Status:** DRAFT — Wave C track 1. Do not commit.  
> **Scope:** §30.4 lever 7 (cross-queen shared read cache) + §30.5 axis 1/2 integration.  
> **Target:** 40–60% cache hit rate on real multi-shard colonies.

---

## 1. Problem statement

In a super-queen colony, child queens independently derive file context (e.g. reading `src/auth/middleware.py` into their context window). When Queen-A and Queen-B both need the same file, the work is duplicated: twice the token burn, twice the wall-clock time. The super-queen is the natural place to centralise this because it already brokers dispatch and sees every shard’s `files_allowed` declaration.

**Hypothesis:** a super-queen-managed read cache with 1-hour TTL can eliminate 40–60% of redundant file reads in colonies where shards share utility code, types, or framework boilerplate.

---

## 2. Architecture

### 2.1 Storage layout

```
~/.claude/state/super-queen-cache/
  .stats.json                     — global telemetry (hits, misses, evictions)
  <sha256(abs-path)>/
    content                       — immutable snapshot of file bytes at cache time
    meta.json                     — {path, sha, cached_at, last_accessed, hits}
```

- **Keying:** SHA-256 of the absolute file path. Collisions are astronomically unlikely; if one occurs it means two files have the same absolute path, which is impossible on the same host.
- **Content snapshot:** a plain `cp` of the source file. Binary-safe; no encoding assumptions.
- **Metadata:** JSON for human readability and easy inspection by `colony-watcher.sh` sweeps.

### 2.2 Lifecycle

```
        ┌─────────────────┐
        │   Child queen   │
        │  needs context  │
        └────────┬────────┘
                 │
                 ▼
        ┌─────────────────┐
        │  cache get <p>  │ ──► SHA match? ──yes──► return content (HIT)
        └────────┬────────┘                           update last_accessed
                 │                                    increment hits
                 │ no
                 ▼
        ┌─────────────────┐
        │   Direct read   │ ──► child queen reads file itself
        └────────┬────────┘
                 │
                 ▼
        ┌─────────────────┐
        │   cache put <p> │ ──► super-queen (or child, advisory) writes entry
        └─────────────────┘
```

**Put path:** triggered by the super-queen after it observes a child queen’s successful file read, or by the child queen opportunistically on first read. Because puts are idempotent and cheap, either pattern works.

**Get path:** triggered by any child queen before expensive context derivation. On miss, the child queen falls back to direct read without blocking.

### 2.3 Integration points

| Protocol component | Integration | Notes |
|---|---|---|
| `colony-watcher.sh` (§29.8) | `sweep-stale` subcommand | Watcher runs `super-queen-cache.sh sweep-stale` every 10 min to reap >1h entries. Adds `CACHE_SWEEP` log events. |
| `dispatch-lock-from-path.sh` (§29.10) | **Not bypassed** | Cache reads are advisory. A child queen must still acquire its dispatch lock before writing. The cache does not grant write authority. |
| PostToolUse hook | `invalidate-by-sha` | When a tool emits `Edit` or `Write` on `<path>`, the hook computes the live file’s new SHA and calls `invalidate-by-sha <old-sha>` + `invalidate <path>` to remove stale entries. |
| `sidecar-health.sh` | `stats` read-only | Health surface can report cache hit rate as a non-blocking diagnostic. |

## 3. Hit-rate measurement (telemetry)

### 3.1 Data collected

`~/.claude/state/super-queen-cache/.stats.json` tracks:

```json
{
  "total_hits": 142,
  "total_misses": 98,
  "total_evicted": 12,
  "total_puts": 154,
  "entries": 42,
  "total_bytes": 3670016,
  "hit_rate_pct": 59.17,
  "miss_rate_pct": 40.83
}
```

- **hit_rate_pct** = `total_hits / (total_hits + total_misses)`
- Counters are append-only except for `total_evicted`.

### 3.2 Colony-level attribution

Global counters tell us whether the cache is healthy overall, but they do not tell us *which* files are hot or *which* colony benefited. To calibrate the 40–60% target, we augment per-colony telemetry:

```
~/.claude/state/colony/<colony-id>/
  _cache-telemetry.json   — { "hits_by_path": { "src/types.ts": 3, ... },
                              "misses_by_path": { "src/main.ts": 1, ... } }
```

This is written by the super-queen (not the cache script) by diffing `cache stats` before and after a colony run. It is optional telemetry; the cache functions without it.

### 3.3 Target calibration plan

| Colony type | Expected hit rate | Why |
|---|---|---|
| Mono-repo with shared types | 55–70% | `src/types.ts`, `package.json`, config files touched by many shards |
| Feature-disjoint microservices | 20–35% | Little file overlap; cache mostly cold |
| Refactoring colony | 60–75% | Same files read repeatedly for analysis, few writes |

**Decision rule:** after n≥3 real super-queen colonies, if measured hit rate < 30%, deprioritise v2.21.0 hardening and absorb the cache into colony-watcher as a passive observation layer only. If measured hit rate > 50%, invest in v2.21.1 (cache pre-warming from `files_allowed` declarations).

## 4. Concurrency model

### 4.1 Advisory file locks

The cache must tolerate N child queens racing on the same entry. We use **per-entry advisory locking** with a fallback chain:

1. **Primary:** `flock -x -w 5 <lockfile>` (Linux, macOS with util-linux, most containers).
2. **Fallback:** `mkdir` spinlock (POSIX-atomic, zero dependencies). 50 retries × 100 ms = 5 s timeout.

**Why not one global lock?** A global lock would serialise all cache operations across queens, creating a bottleneck on colonies with high parallelism. Per-entry locking means Queen-A reading `src/auth.py` does not block Queen-B reading `src/billing.py`.

### 4.2 Read vs write semantics

| Operation | Lock mode | Concurrent? |
|---|---|---|
| `get` | Exclusive (flock -x) on entry | No — but lock scope is one entry, so other entries are unblocked. Lock held only for meta.json read + hit-counter bump (~1 ms). |
| `put` | Exclusive on entry | No — but same single-entry scope. |
| `invalidate` | Exclusive on entry | No — safe because invalidation is rare relative to reads. |
| `evict-if-oversized` | Exclusive per entry being evicted | Serialised by the sorting pass; each `rm -rf` acquires and releases its own lock. |

**Note:** `flock` is advisory. A buggy child queen that ignores the lock file can still read `content` directly. This is acceptable because the cache is advisory — the worst case is a slightly stale read, which is caught by the SHA-mismatch check on the next `get`.

### 4.3 Super-queen as serialisation point

In practice the super-queen is the only actor that calls `put` (it observes file reads from child queen telemetry). Child queens call `get`. This natural separation means write-write races on the same entry are rare. If two child queens both call `put` opportunistically, the second `put` simply overwrites with the same content (idempotent for immutable snapshots).

## 5. Invalidation rules

### 5.1 Automatic (triggered)

| Rule | Trigger | Action |
|---|---|---|
| SHA mismatch | `cache get` detects live file SHA ≠ cached SHA | Delete entry, return miss. Logs `INVALIDATE sha_mismatch`. |
| TTL expiry | Entry `last_accessed` > 1 hour ago | `sweep-stale` deletes entry. Logs `INVALIDATE stale`. |
| Write event | PostToolUse hook sees `Edit` / `Write` | `invalidate <path>` + `invalidate-by-sha <old-sha>`. Logs `INVALIDATE write_hook`. |

### 5.2 Capacity-driven (LRU eviction)

- **Threshold:** 100 MB total cache size.
- **Policy:** LRU by `last_accessed` timestamp. Oldest entries evicted first until total ≤ 100 MB.
- **Measurement:** `stat` on each `content` file; sum before evicting.
- **Graceful degradation:** if disk is too full to write a new entry, `put` skips caching and exits 0 (non-blocking). The child queen has already read the file directly.

### 5.3 Exclusion list (mutable state)

The following are **never cached** (hard reject in `put`):

- `.git/index`, `.git/HEAD`, `.git/logs/*`
- `.env`, `.env.*`
- `/tmp/*`, `*/node_modules/*`, `*/__pycache__/*`
- `~/.claude/state/*` (protocol internal state)

Rationale: these files mutate outside the PostToolUse hook’s visibility, so the cache would silently serve stale data.

## 6. Failure modes and mitigations

### 6.1 Cache corruption (entry unreadable)

**Scenario:** power loss during `put` leaves `meta.json` half-written or `content` truncated.

**Detection:** `cache get` checks that both `meta.json` and `content` exist and that `meta.json` is valid JSON. Any failure is treated as a miss.

**Recovery:** `get` does not auto-repair. `sweep-stale` and `evict-if-oversized` both ignore malformed entries (they just count bytes for the size total). The operator can run:

```bash
find ~/.claude/state/super-queen-cache -name 'meta.json' -exec sh -c '
  python3 -c "import json; json.load(open(\"{}\"))" 2>/dev/null || rm -rf "$(dirname {})"
' \;
```

**Prevention:** `put` writes `content` first, then `meta.json`. A crash after content but before meta leaves an orphan content file with no metadata. This is harmless (ignored on get, swept by size calculations).

### 6.2 Disk full

**Scenario:** cache partition has < 1 MB free. `put` is called.

**Mitigation:** `put` checks `df -k` free space before writing. If insufficient, it logs `[cache] SKIP — disk full` and exits 0. The caller (child queen) already has the file content from its direct read; nothing breaks.

**Monitoring:** `cache stats` reports `total_bytes`. A cron job or `colony-watcher.sh` can alert when `total_bytes > 80 MB` (early warning before the 100 MB hard eviction).

### 6.3 Permission errors

**Scenario:** `~/.claude/state/super-queen-cache/` is owned by a different user (e.g. queen running under `sudo` or a different OS account).

**Mitigation:** All cache operations are best-effort. Permission denied during `put` → skip cache, exit 0. Permission denied during `get` → treat as miss, exit 1. Permission denied during `invalidate` → log to stderr, exit 0 (idempotent).

**Policy:** The cache directory is created with default umask. We do not `chmod 777` because cross-user caching is out of scope for v2.21.0. Multi-host fencing (v3.0) will address this with network-attached storage and ACLs.

### 6.4 Lock starvation

**Scenario:** One child queen holds an entry lock for an extended period (buggy script, debugger attached).

**Mitigation:** Lock timeout is 5 seconds. After timeout, the operation is treated as a skip (`put`) or miss (`get`). The child queen falls back to direct read. No deadlock can block the colony because the cache is advisory.

## 7. Hook integration (PostToolUse)

The PostToolUse hook is not part of this draft (it lives in the queen-runtime layer), but the contract is:

```python
# Pseudocode — queen runtime extension (deferred to v2.21.1)
def on_post_tool_use(tool_name: str, path: str):
    if tool_name in ("Edit", "Write"):
        old_sha = compute_sha_before_edit(path)  # runtime tracks this
        subprocess.run([
            "super-queen-cache.sh", "invalidate", path
        ], capture_output=True)
        subprocess.run([
            "super-queen-cache.sh", "invalidate-by-sha", old_sha
        ], capture_output=True)
```

**Rationale for dual invalidation:** `invalidate <path>` removes the canonical entry. `invalidate-by-sha <old-sha>` catches edge cases where the path was cached under a different absolute path (symlinks, case variations). Both are idempotent and cheap.

## 8. Open questions for v2.21.1

1. **Cache pre-warming:** Should the super-queen call `put` on every file in a shard’s `files_allowed` list *before* dispatch? This would raise hit rate but increase upfront I/O.
2. **Compression:** At 100 MB, text source code compresses ~3–5×. Should we `gzip` `content` to extend capacity? Adds CPU overhead; measure first.
3. **Cross-host sharing:** v3.0 multi-host colonies need a network-attached cache (Redis, NFS, or S3). Keep the current filesystem interface and swap the backend?
4. **Hit-rate telemetry streaming:** Should `cache get` emit structured events to the meshboard (§30.5 axis 3) so the super-queen can show real-time cache effectiveness?

## 9. Acceptance criteria (draft validation)

- [ ] `super-queen-cache.sh put <file>` creates an entry with correct SHA.
- [ ] `super-queen-cache.sh get <file>` returns content on hit, exits 1 on miss.
- [ ] Editing the file then calling `get` invalidates the entry (SHA mismatch).
- [ ] `cache stats` reports hit rate consistent with manual counts.
- [ ] Filling cache to > 100 MB triggers LRU eviction (oldest `last_accessed` removed first).
- [ ] `sweep-stale` removes entries older than 1 hour.
- [ ] Running under low disk space causes graceful skip, not failure.
- [ ] Two concurrent `get` calls on the same entry do not corrupt data.
