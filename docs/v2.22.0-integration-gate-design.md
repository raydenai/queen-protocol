# Colony-level Integration Verification Gate — Design Document

**Protocol version:** v2.22.0 (Wave D track 1)  
**Lever:** §30.4 lever 10 — Cost N→1, quality preserved  
**Status:** DRAFT — do not commit until v2.22.0 milestone review

---

## 1. Problem Statement

In a super-queen colony with N child queens (shards), each child queen currently runs its own integration-test pass at converge time (§2.6 step 8, semantic-conflict probe). When N shards touch overlapping subsystems, the same integration tests run N times against partially-converged intermediate states. This is:

- **Cost-inefficient:** N× pytest/vitest invocations where 1 suffices.
- **State-misleading:** intermediate converge states may fail tests that pass on the final merged tree.
- **Quality-blind:** no single gate explicitly validates cross-shard contracts.

### 1.1 The N×1× trap

| Shards | Current cost (wall-clock proxy) | v2.22.0 cost |
|--------|--------------------------------|--------------|
| 2      | 2 integration passes + semantic probes | 1 gate pass |
| 5      | 5 passes + redundant probes | 1 gate pass |
| 10     | 10 passes (often serialized on shared files) | 1 gate pass |

The colony-level gate runs **after** all child queens have converged into the §2.6 integration worktree. It sees the final merged state — the exact tree that will LAND.

## 2. Integration Surface Detection Algorithm

The gate must determine *what* to test without testing everything. It derives an **integration surface** from the shard graph and the merged worktree.

### 2.1 Inputs

1. `plan.json` — shard definitions with `files_allowed` globs.
2. `shards/<id>/diff.patch` — actual diff produced by each shard.
3. Integration worktree (`colony/<id>/integration/`) — final merged tree.

### 2.2 Detection layers

**Layer A — Cross-shard file overlap (sufficient condition)**

For every file path appearing in any shard diff, count how many shards modified it.

```
∀ file f in ⋃ shard_diffs:
  shard_count(f) = |{ s ∈ shards : f ∈ changed_files(s) }|
  if shard_count(f) ≥ 2:
    f ∈ INTEGRATION_SURFACE
```

These files are *by definition* integration points — two independent ants touched the same file. Even if the merge is textually clean, the semantic interaction is untested by either shard's isolated test run.

**Layer B — Schema migration detection**

Any file matching `.*\.(sql|prisma|alembic)$` in any shard diff is a migration. Migrations are integration points even if only one shard touched them, because downstream code may depend on the schema implicitly.

```
∀ file f in ⋃ shard_diffs:
  if match_schema_pattern(f):
    f ∈ INTEGRATION_SURFACE
    dependents(f) ⊆ INTEGRATION_SURFACE   // heuristic via grep
```

**Layer C — Barrel / facade export detection**

Barrel files re-export symbols from submodules. A change to a barrel's exports breaks every importer, even if the importer file wasn't modified by any shard.

```
∀ file f in ⋃ shard_diffs:
  if match_barrel_pattern(f):   // __init__.py, index.ts, mod.rs, types.ts
    f ∈ INTEGRATION_SURFACE
    dependents(f) ⊆ INTEGRATION_SURFACE   // heuristic via grep
```

**Layer D — Dependent discovery (static heuristic)**

For each Layer B/C file, discover unmodified files in the worktree that reference it:

```bash
grep -r -l --include='*.{py,ts,tsx,js,jsx,rs}' \
  -E "(from .*/<stem>|import .*<stem>|use .*<stem>|mod <stem>)" .
```

Only include dependents that are **not** already in a shard diff — shard-level tests already cover modified files.

### 2.3 Output: Integration Surface

The integration surface is a deduplicated, sorted list of file paths. It is typically:

- 2–15 files for a 3-shard colony
- 0 files for perfectly disjoint shards with no schema/barrel changes

## 3. Test Execution Strategy

### 3.1 What NOT to run

- **Shard-level unit tests:** already passed at child-queen converge time (§2.6 step 4). Re-running them is the N×1× trap.
- **Lint / typecheck:** already passed at §2.7 (verify-done.sh Tier 1A/1B). The integration worktree is lint-clean before this gate fires.

### 3.2 What TO run

- **Integration tests that reference the integration surface.**
  - Python: `pytest` targets that import or string-reference any surface file.
  - TypeScript: `vitest run` targets that import or string-reference any surface file.
- **Contract tests:** round-trip serialization, DB migration up/down, API request/response fixtures.

### 3.3 Scope heuristic

The gate discovers test files by grepping for surface-file stem references inside the worktree's test directories. This is coarse but fast (<1s) and deterministic. False positives (test references file but doesn't test the integration) are harmless — they run and pass. False negatives (integration test uses an indirect reference) are acceptable because the alternative is running the entire test suite, which violates the cost goal.

**Recommendation for repos:** name integration tests after the contract they exercise, e.g. `tests/integration/test_user_schema_events.py` — the stem `user_schema` will match the migration file `migrations/2026_05_20_user_schema.sql` and be picked up by Layer D.

## 4. Failure Modes & Resolution

### 4.1 Integration test fails, all shards pass

This is the **canonical lever-10 value proposition.** It means:

> Two individually-correct changes interact badly in the merged state.

**Protocol behavior:**
- Gate exits 1 (blocked).
- LAND is prohibited.
- Super-queen surfaces the failing integration point(s) with structured stderr.
- Operator options:
  1. **Re-dispatch reconciliation ant** — give the ant both shard diffs + failing test output as `# Prior context:`.
  2. **Revert one shard** — mark it `ABANDONED`, re-converge from pre-shard snapshot.
  3. **Manual fix** — operator edits integration worktree directly, then re-runs gate.

### 4.2 No integration tests exist

**Behavior:** Gate does NOT pass silently. It surfaces a WARNING:

```
NO INTEGRATION TESTS FOUND for detected surface.
Surface contains 4 file(s), but zero test files reference them.
Recommendation: add pytest/vitest integration tests under tests/integration/ ...
```

The gate exits 0 (green) because there is no failing test to block on, but the WARNING is logged to `telemetry.jsonl` and surfaces in the super-queen aggregation report. This creates operator pressure to add coverage without creating a false "everything is fine" signal.

### 4.3 Empty integration surface

If shards are truly disjoint and touch no schemas/barrels, the surface is empty. The gate exits 0 with a soft WARNING recommending cross-shard contract tests if implicit conventions exist (e.g., event bus names, REST route paths).

## 5. Lever 4 — Verification Reuse

Lever 4 (verification reuse) is extended from shard-level reviews to colony-level integration.

### 5.1 Cache key

```
cache_key = SHA256( merge_base_commit + sorted(integration_surface) + test_command )
```

- `merge_base_commit`: the git merge-base of the integration worktree. Ensures cache invalidation on upstream changes.
- `sorted(integration_surface)`: the exact file list. Order-independent via sorting.
- `test_command`: e.g. `pytest tests/integration/`. Captures test-runner changes.

### 5.2 Subset-skip rule

If a prior cached run on the **same merge base** returned **PASS** for a **superset** of the current integration surface, the current run is skipped.

```
∃ cached_run:
  cached_run.merge_base == current.merge_base
  ∧ cached_run.result == PASS
  ∧ current.surface ⊆ cached_run.surface
  → SKIP
```

Why superset-only? If the prior run failed, we don't know which integration point caused the failure — it could be a file *not* in the current surface, or it could be interaction between a removed file and a remaining one. Conservative re-run is required.

### 5.3 Cache storage

`~/.claude/state/colony/.integration-cache/<truncated-sha>.json`

Each entry:
```json
{
  "colony_id": "2026-05-22-feature-x",
  "merge_base": "a1b2c3d4...",
  "result": "PASS",
  "surface": ["src/auth/__init__.py", "migrations/2026_05_20_add_role.sql"],
  "timestamp": "2026-05-22T09:54:00Z"
}
```

Cache GC is handled by the standard `~/.claude/state/colony/` retention policy (§8.3).

## 6. Exit Contract

| Scenario | Exit code | LAND | stderr |
|----------|-----------|------|--------|
| All integration points pass | 0 | Allowed | `["PASS",<file>,""]` per point |
| One+ integration points fail | 1 | Blocked | `["FAIL",<file>,""]` per point + BLOCKED banner |
| No surface, no tests | 0 | Allowed | WARNING + recommendation |
| No surface, tests exist (unlikely) | 0 | Allowed | Tests skipped — nothing to validate |
| Lever-4 cache skip on prior PASS | 0 | Allowed | `["SKIP","lever-4-cache-reuse",<reason>]` |

**Note for Stop-hook adaptation:** If this script is wired as a Claude Code Stop hook (like `verify-done.sh`), change the blocking exit code from `1` to `2` so Claude surfaces stderr back to the agent as a system reminder.

## 7. Test Corpus: 3 Fixture Colonies

See `fixtures/` directory for JSON definitions. Each fixture is a self-contained colony definition with intentional cross-shard regressions.

### 7.1 Fixture A — Schema migration + API drift

**Colony:** `colony-a-schema-api-drift`  
**Shards:** 2  
**Regression:** Shard-1 adds a nullable `phone_verified` column. Shard-2 adds a Pydantic schema that marks `phone_verified` as `bool` (non-nullable). The migration default is `NULL`; the API serializes `None` → Pydantic validation error.

**Integration surface:**
- `migrations/2026_05_20_add_phone_verified.sql` (Layer B)
- `src/schemas/user.py` (Layer C — barrel re-exports `UserCreate`)
- `src/api/routes/users.py` (Layer D — dependent of schema, not in any shard diff)

**Expected gate result:** FAIL on `src/schemas/user.py` — pytest integration test `tests/integration/test_user_registration.py` catches the nullable mismatch.

### 7.2 Fixture B — Cross-shard barrel collision

**Colony:** `colony-b-barrel-collision`  
**Shards:** 3  
**Regression:** Shard-1 exports `formatCurrency` from `lib/i18n/index.ts`. Shard-2 exports `formatCurrency` from `lib/currency/index.ts`. Shard-3 imports from `lib/i18n` but the merged barrel now resolves to the currency version with a different signature (locale string vs locale enum).

**Integration surface:**
- `lib/i18n/index.ts` (Layer C)
- `lib/currency/index.ts` (Layer C)
- `components/PriceDisplay.tsx` (Layer D — dependent)

**Expected gate result:** FAIL on `lib/i18n/index.ts` — vitest `PriceDisplay.test.tsx` renders `$10.00` but gets `10,00 €` due to wrong formatter.

### 7.3 Fixture C — False-positive clean colony

**Colony:** `colony-c-clean`  
**Shards:** 2  
**Scenario:** Shard-1 touches `src/billing/invoice.py`. Shard-2 touches `src/billing/receipt.py`. Both import from `src/billing/__init__.py`, but neither modifies the barrel. No schema changes. No shared files. Existing integration tests pass.

**Integration surface:**
- Empty (no cross-shard files, no migration/barrel modifications)

**Expected gate result:** Green (exit 0) with WARNING: "NO INTEGRATION SURFACE DETECTED — consider adding cross-shard contract tests."

## 8. Anti-fixes (preserved from spec)

1. **Do NOT commit.** This is a draft for v2.22.0 milestone review.
2. **Do NOT re-run shard tests.** The gate's entire value is avoiding redundant shard-level test execution.
3. **Do NOT silently pass on missing tests.** Surface as WARNING to drive coverage growth.
4. **Do NOT run full test suite.** Always scope to the integration surface.
5. **Do NOT cache FAIL results for subset skipping.** Only PASS supersets are eligible for lever-4 reuse.

## 9. Open Questions for Review

1. **Layer D grep heuristic accuracy:** Should we use `ast-grep` / `tree-sitter` instead of regex for import discovery? Trade-off: accuracy vs. dependency on external binaries.
2. **Multi-host colonies:** When sub-queens run on different hosts, the integration worktree may be on the super-queen's host. Is the cache key `merge_base` sufficient, or do we need a host-id salt?
3. **Non-py/ts stacks:** The current script handles Python + TypeScript. Should v2.22.0 include Rust (`cargo test`) and Go (`go test`) integration scoping, or defer to v2.22.x?
4. **Honeycomb broker interaction:** If a colony uses Model R (broker + subscribers), the broker's published types may not appear in any shard diff. Should the broker's `manifest.json` be treated as an implicit Layer C barrel?
