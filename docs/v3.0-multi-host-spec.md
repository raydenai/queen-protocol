# v3.0.x Multi-Host Fencing — Design Spec (Phase Group 7)

Status: SPEC (not implementation). Implementation deferred until all v2.x phases stable + n≥3 production colonies exercise the single-host pattern at scale.

## Why this is a v3 graduation, not a v2.x patch

The protocol is currently **single-host production-ish**. The dispatch-lock, sidecar-health, colony-watcher, sidecar caps — all assume one machine. The shift to multi-host means:

1. Dispatch-lock must be a **distributed** lock (not local file-based)
2. Sidecar caps must be **shared** across machines (one Codex 20/day cap, not 20/day per machine)
3. Worktrees must be **discoverable** by remote queens
4. Mesh state must be **authoritative** (claude-mesh today is signaling; v3.0.x makes it source-of-truth)

These are fundamentally different correctness contracts. Hence v3.

## v3.0.0 — Distributed dispatch-lock

### Current single-host implementation

`~/.claude/scripts/dispatch-lock-from-path.sh` uses local filesystem files:
- Lock file at `<colony>/dispatch.lock`
- Holder PID written to file
- Stale-lock detection via `kill -0 <pid>`

This works for one machine. On multi-host, two queens on different machines both successfully acquire the local lock — race condition.

### v3.0.0 design

Three backend options, ranked by complexity:

#### Option A: Postgres advisory locks (RECOMMENDED)

- `pg_try_advisory_lock(hashtext('<colony>/<shard>'))`
- Single Postgres instance acts as lock coordinator
- Holder identity in shared table: `dispatch_locks(colony, shard, holder_host, holder_pid, acquired_at)`
- Stale-lock detection: per-host heartbeat table; if heartbeat gap >5min, lock can be broken
- Failure mode: Postgres down → fail closed (no queen can dispatch) OR fail open (revert to local lock); operator-configurable

**Pros:** correctness, transactional, mature
**Cons:** requires Postgres dependency (current protocol has no DB requirement)

#### Option B: Redis with Redlock

- `SET <colony>/<shard> <holder> NX EX 300` (atomic SET-if-not-exists with TTL)
- Redlock algorithm for multi-Redis correctness if needed
- Holder identity stored in Redis value
- Stale-lock detection: TTL expiry + holder-side renewal loop

**Pros:** lightweight, no Postgres, Redis is operationally simple
**Cons:** Redlock has known correctness debates; not strictly safe under network partition

#### Option C: ZooKeeper / etcd (consensus-based)

- Ephemeral nodes for lock holding
- Watch-based notification when lock released
- Strong consistency guarantees

**Pros:** strongest correctness
**Cons:** heaviest infrastructure dependency

### Recommendation

Ship v3.0.0 with **Option A (Postgres advisory locks)** as primary, with Option B (Redis) as a configurable backend for low-infrastructure deployments. Reject Option C until ZooKeeper/etcd is already in the operator's infrastructure for other reasons.

### Migration path

1. v3.0.0-rc1: Postgres backend implemented; `dispatch-lock-from-path.sh` gains `--backend=postgres|local` flag, defaults to local
2. v3.0.0-rc2: Operator opts in via `--backend=postgres`; smoke tests on 2 machines
3. v3.0.0: Default flips to postgres if `DISPATCH_LOCK_BACKEND=postgres` env var set
4. v3.0.1 covers cross-host sidecar pool sharing

### Failure modes to test

- Postgres unreachable mid-acquire → fail closed (refuse dispatch), log explicit error
- Postgres reachable but lock holder host unreachable → stale-lock break after heartbeat gap
- Network partition between queen and Postgres → queen refuses dispatch (correctness over availability)
- Lock holder host crashes → heartbeat stops → other queens detect stale-lock after 5min

## v3.0.1 — Cross-host sidecar pool sharing

### Problem

Sidecar caps (Kimi 35/day, Codex 20/day, Gemini 40/day) are PER-API-KEY, not per-machine. Currently each machine reads its own `.daily-cap` file. Two machines using the same API key can each dispatch up to the cap → 2× over-cap.

### v3.0.1 design

- Shared cap counter in Postgres: `lane_usage(lane, date, count)` table
- Each dispatch increments via `UPDATE ... RETURNING count`; refuses dispatch if `count >= cap`
- Telemetry: per-machine usage breakdown for cost attribution

### Failure mode

Postgres unreachable → fall back to local `.daily-cap` (degraded mode, may over-cap). Surface to operator.

## v3.0.2 — Authoritative mesh state via claude-mesh

### Current state

`claude-mesh` (github.com/umitkacar/claude-mesh) is a cross-host **signaling** layer. Queens broadcast "I'm starting shard X"; other queens see the broadcast but it's advisory, not authoritative.

### v3.0.2 design

Promote claude-mesh state to **authoritative**:
- Colony state lives in mesh, not in any one host's filesystem
- Queen lifecycle events (started/converged/blocked) commit to mesh before local action
- Mesh becomes the single source of truth; local files are caches

### Coverage gap

Requires claude-mesh upstream changes (not just protocol work). Coordinate with maintainer before v3.0.2 starts.

## v3.0.x acceptance criteria

- Two queens on different machines: cannot both acquire same shard's lock (correctness)
- Shared cap: total dispatches across machines respects single per-API-key cap
- Mesh: queen B on host 2 can read queen A on host 1's status in <1s of state change
- Failure modes: Postgres-down / network-partition / host-crash all surface explicitly to operator + log, never silently corrupt

## What this version (v3.0.0) does NOT cover

- **Cross-cluster federation** — v3.0.x assumes one cluster (one Postgres / one Redis instance). Multi-cluster is v3.5+ if ever needed.
- **Encryption-at-rest for the lock table** — current `dispatch.lock` files don't encrypt; v3.0.x maintains parity. If lock metadata contains secrets, that's a separate hardening pass.
- **Performance benchmarking** — design targets correctness, not throughput. If lock acquisition becomes a bottleneck (>100 acquires/sec), revisit.

## Open questions for design review

1. Should v3.0.0 require Postgres, or treat it as optional with local fallback? (Recommend: optional, local fallback for solo operators.)
2. How does cap-aware autopilot (v2.19.3) interact with shared caps? (Answer: autopilot reads the shared counter, same logic, different backend.)
3. What if claude-mesh upstream is unwilling to make state authoritative? (Answer: v3.0.2 either forks claude-mesh or builds protocol-internal authoritative mesh.)
