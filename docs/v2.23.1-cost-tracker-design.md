# Queen Protocol v2.23.1 — Cost Ceiling Dashboard (Design Draft)

> Status: DRAFT, NO COMMIT  
> Track: Wave E track 2  
> Depends: v2.19.2 `cap-autopilot.sh`

---

## 1. Goal

Operator-facing visibility into:
- **"How much did this colony cost?"** — per-lane spend breakdown.
- **"Alert if total cost > N dollars"** — configurable ceiling with threshold warnings.
- **"Which lane is blowing the budget?"** — per-lane and per-feature attribution.

This draft intentionally does **not** auto-block dispatches; it surfaces data so the operator (or `cap-autopilot`) can make informed substitution decisions.

---

## 2. First-Cut Cost Model

### 2.1 Per-lane coefficients (operator-tunable)

| Lane | Input / 1M tok | Output / 1M tok | Avg dispatch (USD) | Notes |
|------|----------------|-----------------|--------------------|-------|
| Kimi K2.6 | $0.50 | $2.00 | **$0.10** | |
| Codex GPT-5.5 | $1.25 | $10.00 | **$0.30** | |
| Gemini Flash | $0.075 | $0.30 | **$0.05** | Free OAuth tier = $0 input; paid tier shown |
| Grok | $3.00 | $15.00 | **$0.50** | |
| Gemma 4 local | — | — | **$0.00** | Local inference, unmetered |
| Claude (Claude Code) | — | — | **$0.00** | Subscription model; counted as $0 here, surfaced in cap |

Coefficients live in `~/.claude/state/cost-tracker/coefficients.json`. Operators may override averages or add new lanes without touching source code.

### 2.2 Estimation accuracy

**Problem:** Real cost requires exact input + output token counts per dispatch. Current lane-task scripts (v2.19.x) do **not** record token counts. Adding this is an extension targeted for v2.23.x (see §5 — Future work).

**First-cut mitigation:**
- Use `dispatch count × avg_dispatch_usd` per lane.
- Averages are calibrated from public API pricing + observed typical prompt sizes.
- Accuracy is expected to be within ±30% for interactive coding tasks; acceptable for budgeting, not for invoicing.

### 2.3 Privacy guard

- Token-level data is **never** logged unless the operator explicitly opts in via a future `COST_TRACKER_LOG_TOKENS=1` flag.
- The accumulator stores only: timestamp, lane name, dispatch ID, colony ID, and optional feature tag.

## 3. Telemetry Storage

```
~/.claude/state/cost-tracker/
├── coefficients.json          # Operator-tunable lane prices
├── ceilings.json              # Per-colony USD ceilings
├── colonies/
│   ├── <colony-id>.jsonl      # One line per dispatch
│   └── ...
└── features/
    ├── <feature>.json         # Aggregated cross-colony dispatch list
    └── ...
```

### 3.1 Colony JSONL schema (one line per dispatch)
```json
{"ts":"2026-05-22T17:30:00Z","lane":"kimi","dispatch_id":"abc123","feature":"auth-refactor"}
```

### 3.2 Feature aggregate schema
```json
{
  "dispatches": [
    {"ts":"...","lane":"kimi","dispatch_id":"abc123","colony_id":"auth-queen"}
  ]
}
```

### 3.3 Ceilings schema
```json
{
  "auth-queen": 5.00,
  "api-queen": 10.00
}
```

## 4. CLI Interface

```bash
# Record a dispatch
cost-tracker.sh record <lane> <dispatch-id> [--colony <id>] [--feature <name>]

# Reports
cost-tracker.sh report <colony-id>
cost-tracker.sh report --feature <feature-name>
cost-tracker.sh report --since <iso-date>

# Ceiling management
cost-tracker.sh ceiling --set 5.00 --colony <id>
cost-tracker.sh ceiling --check        # Alert if any colony >80% ceiling
```

### 4.1 Ceiling behavior (warning-only)
- `--check` emits `ALERT: <colony> NEAR-CAP` at ≥80% and `AT-CAP` at ≥100%.
- **Anti-fix:** Never blocks or rejects dispatches. The operator decides whether to throttle, substitute, or accept the overrun.
- Integration point: `cap-autopilot.sh` may read `cost-tracker.sh report <colony> --json` (future wire-up) to factor cost into its `recommend` logic alongside cap-state.

## 5. Integration with cap-autopilot (v2.19.2)

Today `cap-autopilot` decides substitutions based on **dispatch count caps** (e.g., 30 Kimi/day). v2.23.1 adds a **cost dimension**:

- **Cost ceiling + cap together** inform substitution decisions. A lane may have cap headroom but the colony may be near its cost ceiling; conversely, a lane may be near cap but the colony has budget headroom.
- Proposed integration (not in this draft): `cap-autopilot recommend` queries cost-tracker for the target colony and elevates the substitution preference if cost >80% ceiling, even when dispatch count is below cap.
- Telemetry link: `cap-autopilot.log` records substitution events (`kimi → gemini`). `cost-tracker` records lane usage. Cross-referencing both logs gives full "why was this expensive?" forensics.

## 6. Future Work (v2.23.x)

1. **Token-count extension:**
   - Add optional `--input-tokens` and `--output-tokens` to `cost-tracker.sh record`.
   - When present, compute exact cost: `(input/1M × input_rate) + (output/1M × output_rate)` instead of avg dispatch.
   - Requires lane wrappers (`kimi-task`, `codex-task`, etc.) to capture token counts from API responses and forward them.

2. **JSON output mode:**
   - Add `--json` flag to all `report` and `ceiling --check` commands for machine consumption by `cap-autopilot` or dashboards.

3. **Operator opt-in for token logging:**
   - `COST_TRACKER_LOG_TOKENS=1` stores per-dispatch input/output tokens in the JSONL record.
   - Default remains privacy-preserving (no token data).

4. **Retention / compaction:**
   - Colony JSONL files grow unbounded. A future `cost-tracker.sh compact --before <date>` could roll daily records into monthly aggregates.

---

## 7. Anti-fixes (invariants)

| Rule | Rationale |
|------|-----------|
| **Never auto-block dispatches** | Cost is estimated, not exact. Operator retains agency. |
| **Never commit token-level data by default** | Privacy preference; token counts can reveal source code or prompts. |
| **Never mutate `cap-autopilot.sh` in this draft** | Integration is design-only; wire-up is a later PR. |
