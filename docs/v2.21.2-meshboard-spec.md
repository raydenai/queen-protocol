# v2.21.2 — Cross-queen progress aggregation via meshboard: Design Spec

Status: SPEC + skeleton (Claude-authored inline; concurrent-cap blocked Kimi dispatch). Implementation deferred but architecturally complete.

## Goal

Extend the §29.8 colony-watcher daemon to consume status events from N child queens and emit a unified meshboard view to the super-queen. §30.5 axis 3 (progress aggregation) becomes operational.

## Architecture

```
Queen-A writes:                Queen-B writes:                Queen-C writes:
~/.claude/state/queens/        ~/.claude/state/queens/        ~/.claude/state/queens/
  queen-a.status.json            queen-b.status.json            queen-c.status.json
       │                              │                              │
       └──────────────┬───────────────┴──────────────────────────────┘
                      │ (every 2s poll)
                      ▼
              meshboard-aggregator (daemon)
                      │
                      ▼
              ~/.claude/state/super-queen/meshboard.json
                      │
        ┌─────────────┴─────────────┐
        ▼                           ▼
   meshboard-view.sh tail     meshboard-view.sh snapshot
   (live operator stream)     (one-shot JSON or human)
```

## Components

### 1. Queen status emitter — `super-queen-emit.sh`

Helper each child queen invokes whenever its state transitions:

```bash
super-queen-emit.sh state PLAN                       # entering plan step
super-queen-emit.sh state DISPATCH --shard s03       # dispatching shard s03
super-queen-emit.sh state WATCH --progress 2/5       # 2 of 5 shards complete
super-queen-emit.sh state BLOCKED --blocked-on "Queen-A's API contract"
super-queen-emit.sh state LAND                       # done
```

Writes atomically (write to `<file>.tmp` then `mv`) to:
```
~/.claude/state/queens/<queen-id>.status.json
```

Format:
```json
{
  "queen_id": "queen-a",
  "state": "WATCH",
  "current_shard": "s03",
  "progress": "2/5",
  "blocked_on": null,
  "started_at": "2026-05-22T15:30:00Z",
  "updated_at": "2026-05-22T15:34:12Z",
  "pid": 12345
}
```

### 2. Aggregator daemon — `meshboard-aggregator.sh`

Runs in background. Polls every 2s:

```bash
# Pseudocode
while true; do
  queens=()
  for status_file in ~/.claude/state/queens/*.status.json; do
    age=$(file_age_seconds "$status_file")
    if [[ $age -lt 600 ]]; then  # 10min stale threshold
      queens+=("$(cat "$status_file")")
    else
      # Stale queen — mark as DEAD in aggregated state
      queens+=("$(jq '.state = "DEAD"' "$status_file")")
    fi
  done

  jq -n --argjson qs "$(printf '%s\n' "${queens[@]}" | jq -s .)" \
    '{queens: $qs, summary: "...", updated_at: now | todateiso8601}' \
    > ~/.claude/state/super-queen/meshboard.json.tmp
  mv ~/.claude/state/super-queen/meshboard.json.tmp \
     ~/.claude/state/super-queen/meshboard.json

  sleep 2
done
```

Launched via:
```bash
meshboard-aggregator.sh start    # daemonize (nohup, write to ~/.claude/logs/meshboard.log)
meshboard-aggregator.sh stop     # SIGTERM
meshboard-aggregator.sh status   # alive? + last update timestamp
```

### 3. View — `meshboard-view.sh`

```bash
meshboard-view.sh tail       # live operator stream: 2s refresh, like `tail -f`
meshboard-view.sh snapshot   # one-shot human-readable banner
meshboard-view.sh json       # emit current state as JSON (for scripts)
```

Snapshot format:
```
=== Super-Queen Meshboard (updated 2s ago) ===

  Queen-A   WATCH      shard 2/5    file: src/auth/middleware.py
  Queen-B   CONVERGE   3/3 ready    merging worktrees
  Queen-C   BLOCKED    1/4          waiting on Queen-A API contract

  Summary: 2 active, 1 blocked. ETA Queen-C unblocks: depends on Queen-A.
```

## Concurrency + atomicity

- **Atomic writes:** all writes use `<file>.tmp` + `mv` pattern. Readers never see partial files.
- **Stale detection:** queen files older than 10 min (no updates) marked DEAD.
- **Aggregator crash recovery:** launchd/systemd-user restart on exit. State is in files, not memory, so no state loss.
- **No locks:** aggregator only reads; queens only write their own file. No contention.

## Failure modes

| Failure | Behavior |
|---|---|
| Queen crashes mid-status write | Last `.status.json` remains; mtime stale after 10min → marked DEAD |
| Aggregator crashes | Last `meshboard.json` remains valid; restart via supervisor (deferred to operator config) |
| Disk full | Writes fail; meshboard goes stale; operator sees "AGGREGATOR STALLED" in view |
| Queen ID collision (2 queens same ID) | Per-queen file collision; last writer wins. Mitigation: super-queen assigns unique IDs at spawn |
| Status file corrupted JSON | Aggregator skips with warning to ~/.claude/logs/meshboard.log |

## Integration with super-queen role (§30)

In the super-queen's spawn-child-queen step:

```bash
# Super-queen spawns Queen-A
QUEEN_ID="queen-a-$(date +%s)"
super-queen-emit.sh state SPAWNED --queen-id "$QUEEN_ID"  # super-queen records spawn

# Inside child queen's lifecycle, every state transition:
super-queen-emit.sh state PLAN
super-queen-emit.sh state DISPATCH --shard "$SHARD_ID"
# ... etc

# Super-queen polls meshboard every 5s to surface to operator:
meshboard-view.sh json | jq -r '.summary'
```

The operator sees ONE summary line that updates as queens progress. Decomposed work feels like a single progress bar even though N queens are working in parallel.

## What v2.21.2 does NOT do

- Does NOT control queens (it's a VIEW, not a CONTROL surface). Decisions still flow from each queen + super-queen.
- Does NOT poll faster than 1s or slower than 5s (configurability constrained to that range; outside is operator-misconfig).
- Does NOT alert on stuck queens automatically — that's v2.22.x (operational UX).
- Does NOT persist historical state — only current state. Telemetry/retrospective is v3.1.x (learning phase).

## Implementation skeleton (file-level)

```
~/.claude/scripts/super-queen-emit.sh        # 80-100 lines bash
~/.claude/scripts/meshboard-aggregator.sh    # 120-150 lines bash (daemon)
~/.claude/scripts/meshboard-view.sh          # 100-120 lines bash
~/.claude/state/queens/                       # per-queen status files
~/.claude/state/super-queen/meshboard.json    # aggregated state
~/.claude/logs/meshboard.log                  # aggregator log
```

## Acceptance criteria for v2.21.2 ship

- 3 simulated child queens emit status events; aggregator produces correct meshboard.json within 4s of last event
- Stale queen (>10min no update) shows DEAD; doesn't block aggregation
- Crash + restart of aggregator: meshboard.json eventually consistent with current queen files
- Operator view (`tail` mode) updates every 2s without flicker
- All writes atomic; no partial JSON observed under concurrent reads
