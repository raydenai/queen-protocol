# Queen Protocol v2.21.1 — Shared-file serialization detector (design draft)

> **Status:** DRAFT — Wave C track 2  
> **Scope:** §30.3 anti-decomposition rule enforcement + §30.5 cross-queen coordination axis 2  
> **Draft location:** `/tmp/v2.21.1-draft/`  
> **Target integration:** between DECOMPOSE and DISPATCH in super-queen flow

---

## 1. Problem statement

The super-queen decomposes user goals into N child queens. Each queen proposes a shard graph with `files_allowed` per shard. When two queens' `files_allowed` sets intersect, they will conflict at converge. The current protocol (§30.3) states this as a HARD rule — "merge them into one shard group with sub-shards" — but provides no automated detection or enforcement. This draft closes that gap.

Two distinct actions are required:

1. **MERGE** — Overlapping queens become one queen with sub-shards. Appropriate when the overlap is structural (lock files, manifests, shared types).
2. **SERIALIZE** — Queens remain separate but run in order, with later queens rebased on earlier queens' output. Appropriate when the overlap is a feature-export point (`__init__.py`, `index.ts`) or test/documentation surface.

A third category is a **hard block** — some overlaps must NEVER happen (migrations, secrets, CI config) and require manual redesign.

---

## 2. Heuristics

### 2.1 ALWAYS MERGE — structural / dependency-management overlaps

These files are "merge-only" because parallel modification without semantic coordination is almost always incorrect. Two queens adding different deps to `package.json` independently will produce a nonsensical merge.

| Category | Patterns | Rationale |
|---|---|---|
| Lock files | `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `poetry.lock`, `uv.lock`, `Cargo.lock`, `composer.lock`, `Gemfile.lock`, `*.lockb` | Lock files are deterministic serialization of a single dependency state. Parallel writes are structurally invalid. |
| Manifests | `package.json`, `pyproject.toml`, `Cargo.toml`, `composer.json`, `Gemfile`, `requirements*.txt` | Dependency declarations, scripts, and metadata must be coherent as a single document. |
| Type exports | `*/types.ts`, `*/types/*.ts`, `*/schemas/*.py`, `*/models/__init__.py`, `*/index.ts`, `*/lib/index.ts` | Shared contract surfaces. Two queens adding types to the same barrel file need single-queen coherence. |
| Shared config | `tsconfig*.json`, `vite.config.*`, `next.config.*`, `tailwind.config.*`, `eslint.config.*`, `*.config.{js,ts}` | Build/tooling configuration is global state; parallel edits produce incompatible environments. |

**Merge semantics:** Combine all shards from both queens into a single synthetic queen. Deduplicate `files_allowed`. Preserve original shard IDs as `sub_shards` for traceability. Priority inherits `critical` if any member was critical.

### 2.2 CAN SERIALIZE — feature-export / test / doc overlaps

These files CAN be serialized when the overlap is incidental — two features both need to register in a barrel file, or both add tests to a shared test directory.

| Category | Patterns | Rationale |
|---|---|---|
| Test files | `*.test.{ts,tsx,py}`, `*_test.py`, `*.spec.{ts,tsx,py}` | Tests are additive. Queen-A adds tests for feature A; Queen-B adds tests for feature B. Serialization ensures test file append ordering, not semantic merge. |
| Documentation | `*.md`, `*.mdx`, `*.rst` | Doc edits are usually additive (new sections). Serialization avoids markdown merge noise. |
| Feature exports | `__init__.py`, `index.ts`, `index.tsx`, `index.js` | Barrel files that re-export feature modules. Each queen adds its own re-export; ordering matters only for human readability. |
| Story files | `*.stories.{tsx,ts}` | Additive UI documentation; same rationale as tests. |

**Serialization semantics:** Build a DAG of blocker relationships. Queen-B blocks on Queen-A for file `F`. Super-queen dispatches Queen-A first, waits for MERGED, rebases Queen-B's worktree, then dispatches Queen-B. The rebase step is mandatory — Queen-B must see Queen-A's changes before editing the shared file.

### 2.3 NEVER OVERLAP — hard block requiring manual redesign

These overlaps are protocol violations. The detector exits with code 3 and refuses any `--fix-*` auto-resolution.

| Category | Patterns | Rationale |
|---|---|---|
| Migrations | `*/migrations/*`, `*/alembic/versions/*`, `*/supabase/migrations/*`, `*/prisma/migrations/*`, `*/db/migrate/*` | Migration ordering is load-bearing. Two queens adding migrations to the same directory must be sequenced by human judgment (timestamp prefix collision, dependency chains). |
| Secrets / keys | `.env*`, `*.key`, `*.pem` | Security-sensitive. Parallel modification of secrets is inherently suspicious. |
| CI config | `.github/workflows/*` | CI changes have blast radius across all queens; should be owned by a single senior shard. |

**Action on detection:** Emit explicit error, list offending queens and files, exit 3. Super-queen refuses dispatch. Operator must redesign.

### 2.4 Default / ambiguous overlaps

Any source file not matching the tables above (e.g., two queens both editing `apps/api/src/main.py`) is treated as **MERGE** with a warning. This is conservative: overlapping source files usually indicate the shard decomposition was wrong — the features are not file-disjoint and should not have been split across queens.

> **Exception note:** Some "shared" overlaps (e.g., shared schema types that need semantic coordination, not just write-ordering) are flagged in the report as requiring human review even when auto-MERGE is applied. The `--fix-merge` output includes a `merge_reason` field for this.

## 3. Integration into super-queen lifecycle

### 3.1 Hook point

The detector runs **between DECOMPOSE and DISPATCH**, after the super-queen has produced child-queen shard graphs but before any worker is spawned.

```
DECOMPOSE → [detect-overlap.sh] → (decision gate) → DISPATCH
```

### 3.2 Decision gate

```bash
# Inside super-queen dispatch logic
detect-overlap.sh --quiet plan.json
ec=$?

case $ec in
  0)  # clean — proceed to dispatch
      dispatch_all
      ;;
  1)  # overlaps exist — auto-fix or human gate
      if [[ "$AUTO_FIX" == "merge" ]]; then
        plan.json=$(detect-overlap.sh --fix-merge plan.json)
      elif [[ "$AUTO_FIX" == "serialize" ]]; then
        serialize_plan=$(detect-overlap.sh --fix-serialize plan.json)
        # super-queen uses serialize_plan to order dispatch waves
      else
        # Surface overlaps to user; require explicit resolution
        surface_plan_checkpoint
      fi
      ;;
  3)  # NEVER overlap — hard stop
      echo "Refusing dispatch: overlapping migrations/secrets detected." >&2
      exit 1
      ;;
esac
```

### 3.3 Refuse to dispatch without `--fix` resolution

By default (`--report`), the detector exits 1 when overlaps exist. The super-queen treats this as a blocking gate. Dispatch proceeds only when:
- No overlaps (exit 0), OR
- Operator explicitly opts into `--fix-merge` or `--fix-serialize`, OR
- Operator manually redesigns the shard graph to eliminate overlaps

This is the §30.3 anti-decomposition rule made enforceable.

## 4. Test corpus

Five fixture shard graphs exercise all paths:

### 4.1 `merge-lockfiles.json` — Two queens both touch `package.json`

- **Queen-A** shard adds a React component (touches `apps/web/src/Button.tsx`, `package.json` for dep)
- **Queen-B** shard adds a util library (touches `apps/web/src/utils.ts`, `package.json` for dep)
- **Expected:** MERGE. Both queens become one merged group because `package.json` is structural.

### 4.2 `merge-types.json` — Two queens both touch shared type barrel

- **Queen-A** adds `UserProfile` type to `packages/types/src/index.ts`
- **Queen-B** adds `BillingEvent` type to same `packages/types/src/index.ts`
- **Expected:** MERGE. Type barrel is a shared contract surface.

### 4.3 `serialize-features.json` — Feature-export overlap on `__init__.py`

- **Queen-A** adds `auth` module to `apps/api/src/services/__init__.py`
- **Queen-B** adds `billing` module to same `__init__.py`
- **Queens are otherwise file-disjoint** (auth logic vs billing logic in separate files)
- **Expected:** SERIALIZE. Auth queen runs first; billing queen rebases then runs.

### 4.4 `no-overlap.json` — Completely disjoint queens

- **Queen-A** touches `apps/web/**/*.tsx`
- **Queen-B** touches `apps/api/**/*.py`
- **Expected:** No overlaps. Exit 0. Clean dispatch.

### 4.5 `migration-conflict.json` — Both queens touch migrations

- **Queen-A** adds `supabase/migrations/20260507_add_users.sql`
- **Queen-B** adds `supabase/migrations/20260507_add_orders.sql`
- **Expected:** NEVER overlap. Exit 3. Super-queen hard-blocks dispatch.

## 5. Future work (post-v2.21.1)

1. **Concrete file manifest integration** — If the repo provides a `files_manifest` (listing of all concrete files), overlap detection becomes precise instead of glob-conservative. The script already accepts this field but does not yet generate it.
2. **Semantic coordination detector** — Some overlaps (shared schema types with cross-references) need more than MERGE; they need a broker ant (§13). Flag these for specialist routing.
3. **Auto-rebase wiring** — `--fix-serialize` currently emits a plan; the super-queen must still implement the rebase. Future version could emit a script that performs the rebase automatically.
4. **Metrics integration** — Emit telemetry events to `telemetry.jsonl` when overlaps are detected and how they were resolved, for retrospective tuning of the heuristic tables.

---

## 6. Anti-fixes (restatement from spec)

- **Do NOT commit.** This is a draft for review; integration happens in v2.21.1 proper.
- **Do NOT auto-fix without explicit `--fix-*` flag.** Report-only by default preserves operator agency.
- **Do NOT assume file-level locks are sufficient.** The detector explicitly flags semantic-coordination overlaps that need broker ants, not just serialization.
