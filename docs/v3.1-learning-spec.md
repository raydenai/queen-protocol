# v3.1.x Quality from Learning — Evidence Collection Plan (Phase Group 8)

Status: EVIDENCE-COLLECTION PLAN (not implementation). v3.1.x is the only phase group that **cannot ship as code without prerequisite data**. Implementation is blocked until n≥10 production colonies feed telemetry. This document specifies what telemetry to collect and how.

## Why v3.1.x is data-bound

v3.1.0 (retrospective-driven matrix tuning), v3.1.1 (lane substitution learning), and v3.1.2 (decomposition pattern library) all require **decisions grounded in real-colony evidence**, not theory:

- v3.1.0: auto-adjust dispatch tier thresholds based on actual win/loss data per shard class
- v3.1.1: learn which lane wins which shard class (Kimi vs Codex vs Gemini for migration / UI / refactor / etc.)
- v3.1.2: proven shard graphs become reusable templates

Without telemetry, all three are guesses. With telemetry, each becomes a data-driven recommendation.

## Telemetry storage

Where the data lives:

- Per-shard: `~/.claude/logs/telemetry/shards/<colony>-<shard>.jsonl` (append-only)
- Per-colony: `~/.claude/logs/telemetry/colonies/<colony>.json` (one per colony)
- Per-substitution: `~/.claude/logs/telemetry/substitutions.jsonl` (append-only)

All local. No telemetry upload. Operator can rotate/archive at their discretion.

## v3.1.0 — Retrospective-driven matrix tuning

### Goal

After n≥30 shards per kind × tier combination, statistically recommend matrix threshold adjustments.

### Algorithm

For each `(kind, tier)` combination:

- Compute win rate (shipped without review-round retry)
- Compute median wall-clock
- Compare against alternative tier for same shard kind (e.g. solo vs review for sub-30-LOC fixes)
- Recommend tier change if alternative wins on BOTH win-rate AND wall-clock

Output: a recommended matrix v3 with empirically grounded thresholds.

### Acceptance criteria

- ≥80% of recommendations align with operator-reported intuitions
- Recommendations come with confidence intervals + sample sizes
- Operator can accept/reject per-rule, not all-or-nothing

## v3.1.1 — Lane substitution learning

### Goal

Per shard kind, rank lanes by historical win rate. The super-queen consults this ranking when routing.

### Algorithm

For each `(kind, lane)` combination:

- Win rate: shipped-first-try / total-dispatched
- Wall-clock median
- Cost per win (median cost / win rate)

Output: a ranked routing table per shard kind, replacing or augmenting the v2.x matrix.

### Example output

```
kind=migration:
  1. codex      win=0.91  median=180s  cost-per-win=$0.15
  2. claude     win=0.88  median=210s  cost-per-win=$0.40
  3. kimi       win=0.72  median=120s  cost-per-win=$0.05

kind=ui-polish:
  1. codex      win=0.95  median=90s   cost-per-win=$0.08
  2. claude     win=0.81  median=120s  cost-per-win=$0.20

kind=test-hardening:
  1. kimi       win=0.89  median=150s  cost-per-win=$0.03
  2. claude     win=0.83  median=200s  cost-per-win=$0.15
```

The super-queen consults this table at routing time instead of the static §1.1 / §4.2 rules.

### Acceptance criteria

- n≥30 per `(kind, lane)` cell before promotion (avoid n=1 anecdotes)
- Recommendations versioned (old static rules preserved as fallback)
- Re-rank after every 100 new colonies

## v3.1.2 — Decomposition pattern library

### Goal

Proven shard graphs that decompose features successfully become reusable templates the super-queen pulls from.

### Algorithm

After n≥3 successful colonies match a decomposition shape:

- Extract: `(feature_signature, shard_count, shard_kinds, dependency_edges, files_allowed_pattern)`
- Save as template: `~/projects/queen-protocol/lib/decompose-patterns/<pattern-id>.json`
- Super-queen consults patterns first; falls back to fresh decomposition if no match

### Example pattern

```json
{
  "pattern_id": "auth-flow-v1",
  "matches": ["build user auth", "implement signin/signup", "add JWT sessions"],
  "shards": [
    {"id": "schema", "kind": "migration", "complexity": 3, "files": ["migrations/*.sql"]},
    {"id": "routes", "kind": "feature", "complexity": 5, "files": ["src/auth/routes.*"], "depends_on": ["schema"]},
    {"id": "middleware", "kind": "feature", "complexity": 4, "files": ["src/auth/middleware.*"], "depends_on": ["routes"]},
    {"id": "tests", "kind": "test", "complexity": 3, "files": ["tests/auth/*"], "depends_on": ["middleware"]}
  ],
  "n_successful_uses": 3,
  "median_wall_clock_minutes": 12
}
```

### Acceptance criteria

- Pattern matches reduce decomposition time >50% vs cold reasoning
- Pattern outcomes match historical median (within 1 SD)
- Library size capped (e.g. 50 patterns); least-recently-used eviction

## What v3.1.x does NOT do

- No telemetry uploaded to any external service. All local, operator-controlled.
- No automatic matrix changes without operator approval. Recommendations only.
- No PII in telemetry — shard IDs, file paths, kinds only. Diff content NOT logged.
- No retroactive re-routing of in-flight shards based on new learning. Learning applies to NEXT dispatch, not current.
