# v2.20.2 — Speculative Dispatch at PLAN: Design Spec

Status: SPEC (not implementation). v2.20.2 extends the queen cycle (§2: Survey → Plan → Dispatch → Watch → Reap → Converge → Verify → Land) to fire high-confidence sub-shards in parallel WHILE plan finalizes. Discard-on-plan-change semantics.

## Why speculate

Today's queen cycle has a serial bottleneck: PLAN must complete before DISPATCH starts. For multi-shard colonies where 70-80% of shards are obvious at first read (e.g. "add export to `__init__.py` after creating new module"), waiting for full plan finalization burns wall-clock that could overlap implementation.

Empirical from existing colonies: PLAN step averages 90-180s; obvious sub-shard implementation is 30-90s each. Speculation overlap could reclaim 60-120s per colony — 30-50% of plan-grain wall-clock.

## State-machine extension (additions to §2.9 shard states)

New states:

```
SPECULATIVE_DISPATCHED  — shard dispatched before plan finalized
                          (may be discarded if plan changes)
SPECULATIVE_CONFIRMED   — plan finalized; this shard kept
SPECULATIVE_DISCARDED   — plan changed; this shard's work is throwaway
                          (worktree cleaned up, no merge)
```

Transitions:

```
DRAFT_PROPOSED → SPECULATIVE_DISPATCHED (queen fires before plan finalize)
              → CONFIRMED                (queen waits for plan finalize, then normal flow)

SPECULATIVE_DISPATCHED → SPECULATIVE_CONFIRMED (plan finalized; matches dispatched shard)
                      → SPECULATIVE_DISCARDED  (plan finalized; shard not in final graph)

SPECULATIVE_CONFIRMED → IN_FLIGHT (continues normal state machine from here)
SPECULATIVE_DISCARDED → ABANDONED (worktree cleanup; no merge)
```

## Confidence threshold — what makes a sub-shard speculatable

A sub-shard is "high-confidence enough to speculate" when ANY of:

1. **Operator named the file/function explicitly** — e.g. "fix the auth middleware bug" → speculate on the auth middleware shard.
2. **Stack-trace file in PLAN context** — error message points to specific line; that shard is high-confidence.
3. **Obvious pattern shard** — adding `__init__.py` exports, generating types from schema, updating CHANGELOG headers. Patterns with >0.9 historical match rate.
4. **Decomposition emitted with `confidence > 0.85`** — the v2.20.0 decomposer assigns confidence per shard; high-confidence shards speculate.
5. **Operator explicit `--speculate <shard-id>` flag** — manual override.

## Plan-change detection (when speculation should be discarded)

The queen monitors for these events during PLAN execution:

1. **User adds constraint mid-plan** — e.g. "oh wait, also make it work with SSO" → re-decompose; any speculative shard whose `files_allowed` overlaps with new scope MUST be discarded (the shard's worktree may have made now-invalid assumptions).
2. **Survey re-runs** — if a follow-up SURVEY step produces a different shard graph (file set, dependency edges), speculatives that don't appear in the new graph are discarded.
3. **Higher-priority shard preempts** — a critical shard surfaces that requires the queen's attention; speculatives that aren't urgent can be paused.
4. **Cap exhaustion** — if speculative dispatch causes paid-lane cap exhaustion, future-confirmed shards have no lane → cancel ongoing speculatives.

## Cost accounting (the make-or-break metric)

Speculative work that's discarded is SUNK COST. Speculation is net-positive iff:

```
expected_value = P(confirmed) × wall_clock_savings - P(discarded) × discard_cost
              > 0
```

Where:
- `P(confirmed)` = probability the speculative shard appears in final plan (depends on confidence threshold)
- `wall_clock_savings` = overlap time gained (~30-90s per shard)
- `P(discarded)` = 1 - P(confirmed)
- `discard_cost` = wall-clock spent on discarded work + worktree cleanup overhead

**Target discard rate:** <15%. Beyond that, speculation is net-negative.

Per-lane sunk-cost budget:
- Gemma 4 local ($0): unlimited speculation (no $ sunk cost)
- Kimi cheap: high speculation OK
- Codex expensive: speculate sparingly, high-confidence only
- Claude (the queen itself): NO speculation — the queen's own work is sequential by design

## Implementation skeleton

In the queen orchestrator (wherever the §2 cycle is encoded — currently distributed across Claude's internal logic + `~/.claude/scripts/` helpers):

```python
# Pseudocode for speculative dispatch at PLAN

class QueenCycle:
    def plan(self, feature_request):
        initial_decomposition = self.decompose(feature_request)  # v2.20.0 template

        # Identify speculatable sub-shards
        speculatable = [s for s in initial_decomposition
                       if s.confidence > SPECULATION_THRESHOLD]

        # Fire speculatives in parallel
        spec_handles = []
        for shard in speculatable:
            handle = self.dispatch(shard, state=SPECULATIVE_DISPATCHED)
            spec_handles.append((shard, handle))

        # Finalize plan (may modify shard graph)
        final_decomposition = self.finalize_plan(initial_decomposition)

        # Confirm or discard speculatives
        for (orig_shard, handle) in spec_handles:
            if self.shard_matches(orig_shard, final_decomposition):
                self.transition(handle, SPECULATIVE_CONFIRMED)
            else:
                self.discard_speculative(handle)
                self.log_telemetry(shard=orig_shard, outcome='discarded', sunk_cost=handle.wall_clock_used)

        # Continue normal dispatch for non-speculative shards
        for shard in final_decomposition:
            if shard.state == CONFIRMED:  # not already in speculative path
                self.dispatch(shard)
```

In `kimi-task.sh` / `codex-task.sh`: support a `--speculative` flag on `start` that marks the metadata file. `result <pid>` can then report the speculative status. `cancel <pid>` is used for discard.

In `colony-watcher` (§29.8): observe SPECULATIVE_DISPATCHED state transitions; alert if discard rate exceeds 25% over a moving window.

## Risk catalog (5+ ways speculation backfires)

1. **Race condition: speculative shard writes A; plan finalizes saying "actually don't write A"** — has to be rolled back. Mitigation: speculative writes go to isolated worktrees only (never directly to main colony tree); discard = `rm -rf <worktree>`.
2. **Worktree pollution** — discarded speculatives leave orphan worktrees. Mitigation: cleanup on transition to SPECULATIVE_DISCARDED + periodic colony-watcher sweep for stale speculative worktrees.
3. **Reviewer noise** — speculative drafts get parallel-reviewed before being discarded; review cycles wasted. Mitigation: skip parallel review for SPECULATIVE_DISPATCHED state; only review on transition to SPECULATIVE_CONFIRMED.
4. **Cap exhaustion** — speculation eats paid lane cap that confirmed work needs. Mitigation: cap-aware autopilot (v2.19.3 → renumbered v2.19.2 in actual ship) treats speculative dispatches as deferrable; cap-deferred dispatches block until cap frees.
5. **Premature converge** — plan changes after speculative converged into main. Mitigation: speculative converge is deferred until SPECULATIVE_CONFIRMED transition; converge only happens after plan finalizes.
6. **Speculation cascade** — speculative shard A spawns sub-speculative B; if A discards, B is now orphan speculation. Mitigation: speculation depth capped at 1 (sub-shards don't speculate).
7. **Discard-rate runaway** — operator's vague prompts cause high plan churn → high discard rate → cost amplification. Mitigation: discard rate >25% over 10 colonies → auto-disable speculation; require operator explicit re-enable.

## Acceptance criteria for v2.20.2 ship

- Speculation produces measurable wall-clock savings ≥20% on a fixture corpus of 5 multi-shard colonies
- Discard rate ≤15% on same corpus
- No silent data loss: discarded speculatives leave NO state in main colony tree (full rollback verified)
- Cap-aware integration: speculative dispatches respect autopilot lane substitution

## What v2.20.2 does NOT do

- Does NOT speculate Claude's own work. Queen-side reasoning is sequential.
- Does NOT speculate without explicit confidence threshold. Random speculation is anti-pattern (cap waste).
- Does NOT auto-tune SPECULATION_THRESHOLD. Manual operator config until v3.1.0 retrospective tuning has data.

## Open questions for design review

1. Should SPECULATION_THRESHOLD be per-shard-kind (e.g. lower for refactors, higher for migrations)? Probably YES, but defer until v3.1.0 evidence.
2. How does speculation interact with parallel review threshold (v2.18.0 matrix: 2+ files → parallel review)? Specifically: should a speculative shard's review fire concurrently with the speculation, or only after confirmation?
3. Cross-queen speculation: in v2.21+ (cross-queen coordination), can Queen A speculate while Queen B's plan finalizes? Defer to v2.21.0 design.
