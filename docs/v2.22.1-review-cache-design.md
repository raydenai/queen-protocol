## 1. Goal

If reviewer-A passed diff X, and shard-B's diff is a strict subset of X, skip shard-B's redundant review. Preserve quality, reduce cost and wall-clock time.

---

## 2. Diff-Subset Detection Algorithm

### 2.1 Hunk decomposition

A unified diff is decomposed into individual hunks. Each hunk is defined by:

1. **File path** — from the `+++ b/<path>` header (new-file side).
2. **Line range** — the `@@ -old_start,old_count +new_start,new_count @@` header.
3. **Content** — every line belonging to that hunk (including the `@@` line itself, plus all context, addition, and removal lines).

Hunks are parsed sequentially; a new `diff --git` block or a new `@@` header terminates the preceding hunk.

### 2.2 Hunk hash

Each hunk is hashed independently:

```
hunk_hash = SHA256(file_path + "\n" + line_range + "\n" + hunk_content)
```

This hash is **position-sensitive** — the same textual change at a different line offset produces a different hash. This is intentional and conservative.

### 2.3 Cache entry

Per hunk, the cache stores a JSON file at:

```
~/.claude/state/super-queen-review-cache/<hunk_hash>.json
```

Schema:

```json
{
  "hunk_hash": "sha256:abc123...",
  "file_path": "src/main.py",
  "line_range": "@@ -42,7 +42,9 @@",
  "passing_reviewers": ["kimi-rescue", "codex-rescue"],
  "recorded_at": "2026-05-22T09:55:00Z",
  "file_mtime_at_record": 1716372900.0,
  "last_accessed_at": "2026-05-22T10:00:00Z",
  "access_count": 3
}
```

- `passing_reviewers` — append-only list of reviewer IDs whose verdict was `PASS` for this exact hunk.
- `file_mtime_at_record` — the file's `mtime` at the moment the review was cached. Used for invalidation.
- `last_accessed_at` — updated on every `check` hit, drives LRU eviction.

### 2.4 Subset check

For a candidate diff to be declared a **SUBSET HIT**, **ALL** of its hunks must satisfy:

1. A cache entry exists for the hunk's hash.
2. The entry is younger than the 7-day TTL.
3. The entry's `passing_reviewers` list is non-empty.
4. The target file's current `mtime` exactly matches `file_mtime_at_record`.

If **any** hunk misses, the entire diff is a **MISS**.

On hit, the script prints:

```
HIT reviewer=<representative-reviewer-id>
```

The first passing reviewer found is returned as the representative; the caller may use it for attribution or telemetry.

### 2.5 Why exact match (no fuzzy)

Fuzzy hunk matching (e.g., ignoring line numbers, using `git diff --patience`, or `difflib` ratio) increases hit rate but introduces **false-positive reuse**: the same code text may behave differently if surrounding declarations, imports, or control flow changed. Because verification reuse skips a mandatory review gate, a false positive ships un-reviewed code.

**Mitigation:** exact-match hashing plus `mtime` gating. The hash captures line range; the `mtime` captures file change. Both must agree.

## 3. Storage & Eviction

### 3.1 Directory layout

```
~/.claude/state/super-queen-review-cache/
  .stats.json              # telemetry sink
  <hunk_hash_1>.json       # per-hunk entry
  <hunk_hash_2>.json
  ...
```

### 3.2 LRU eviction

- **Capacity:** 1,000 hunk entries.
- **Policy:** When a `record` operation pushes the directory above 1,000 non-stats JSON files, evict the entries with the oldest `last_accessed_at` until the count is ≤ 1,000.
- **Rationale:** 1,000 hunks ≈ 100–200 typical colony diffs. At the expected colony velocity, this is roughly 2–3 weeks of working set.

### 3.3 TTL

- **Duration:** 7 days (604,800 seconds).
- **Enforcement:** Checked at `check` time. Expired entries are deleted on access, not by a background sweeper.
- **Rationale:** Diffs rot. A hunk that was safe a month ago may be unsafe after adjacent refactors changed semantics. Seven days balances hit rate against staleness risk.

## 4. Cache Invalidation

### 4.1 File-write invalidation

When the queen (or any ant) performs a `PostToolUse Edit` or `Write` on a cached file, **all hunks touching that file must be invalidated immediately**.

Mechanism:

```bash
review-cache.sh invalidate <file-path>
```

This resolves the file to an absolute path, scans all cache entries, and deletes those whose `file_path` matches.

### 4.2 Where to wire the invalidation hook

- **Queen direct edits:** The queen's tool-use wrapper calls `review-cache.sh invalidate` immediately after a successful `Edit` or `Write`.
- **Ant edits (write-mode shards):** After converge, when the queen applies a shard diff to the integration worktree, iterate over `files_touched` and invalidate each before the next shard's `check` is attempted.
- **External edits (operator, IDE, `git pull`):** These are not observed by the protocol. They are caught defensively by the `file_mtime_at_record` check on the next `check` call.

### 4.3 Why not branch-aware invalidation

Git branch switches and rebases change line numbers, which naturally change hunk hashes. Exact-match hashing means old-branch hunks simply won't match new-branch diffs — no explicit branch invalidation is required. This is a feature, not a bug: it prevents cross-branch false positives.

## 5. False-Positive Risk Analysis

### 5.1 Threat model

| Risk | Scenario | Likelihood | Impact |
|---|---|---|---|
| **Surrounding code changed** | Hunk matches exactly, but a function called within the hunk's scope was modified elsewhere. | Medium | High (unreviewed change ships) |
| **Import/context drift** | Hunk edits a line that references a symbol whose type/contract changed in another file. | Medium | High |
| **Same hunk, different intent** | Two developers independently write identical diff hunks for different reasons (e.g., formatting). | Low | Medium |
| **mtime collision** | File `mtime` unchanged despite content change (rare on modern filesystems; possible with `touch -r`). | Very low | High |

### 5.2 Mitigations

1. **Exact-match hashing** — Line-range + content must match byte-for-byte. No fuzzy tolerance.
2. **File `mtime` gating** — If the file was touched after the review was cached, the hunk is treated as a miss and deleted.
3. **TTL (7 days)** — Limits the window during which a stale hunk can accidentally hit.
4. **Only PASS verdicts cached** — A cached entry means "this hunk was reviewed and approved." FAIL verdicts are never stored, so a previously-failed hunk cannot be skipped.
5. **Conservative by default** — The script errs toward MISS. Any ambiguity (parse error, unreadable cache file, missing mtime) falls through to MISS.

### 5.3 Residual risk

If a file's `mtime` is unchanged but its content is changed (e.g., via `touch -r` or sub-second edits on a low-resolution filesystem), the cache may falsely hit. This is accepted as a low-probability edge case; the protocol's broader gate system (queen-rerun tests, post-converge verify) still catches most semantic issues.

## 6. Telemetry

### 6.1 Stats file

```
~/.claude/state/super-queen-review-cache/.stats.json
```

Schema:

```json
{
  "total_hits": 47,
  "total_misses": 153,
  "total_puts": 312,
  "total_evicted": 18,
  "reviewer_stats": {
    "kimi-rescue": { "hits": 22, "misses": 70 },
    "codex-rescue": { "hits": 25, "misses": 83 }
  },
  "estimated_cost_saved_usd": 2.35
}
```

### 6.2 Metrics collected

| Metric | Source | Use |
|---|---|---|
| `total_hits` | Incremented on every `check` HIT | Aggregate reuse count |
| `total_misses` | Incremented on every `check` MISS | Aggregate miss count |
| `hit_rate` | `hits / (hits + misses)` | Quality of the reuse window |
| `total_puts` | Incremented on every hunk stored during `record` | Cache write volume |
| `total_evicted` | Incremented on LRU eviction | Working-set pressure |
| `per-reviewer hits/misses` | Broken down by representative reviewer on HIT | Lane-level calibration |
| `estimated_cost_saved_usd` | `hits × $0.05` (mid-range review cost) | Rough ROI for operator dashboards |

### 6.3 Cost model

- $0.05 per skipped review is a blended average of Kimi (~$0.03) and Codex (~$0.05–0.08) review costs.
- This is an **estimate**, not an accounting truth. It exists to answer "is this lever worth keeping?"
- If real colony data shows a different marginal cost, update `COST_PER_REVIEW_USD` in `review-cache-core.py`.

### 6.4 Integration with colony telemetry

At colony LAND, the queen should append a `REVIEW_CACHE` event to `<colony-id>/log/telemetry.jsonl`:

```json
{"event": "REVIEW_CACHE", "hits": 3, "misses": 1, "saved_usd": 0.15, "timestamp": "..."}
```

This allows cross-colony trend analysis (e.g., "hit rate improving as codebase stabilizes").

## 7. Interface Contract

### 7.1 `review-cache.sh check <diff-file>`

- **Input:** Path to a unified diff (e.g., `shards/s04/diff.patch`).
- **Output (stdout):** `HIT reviewer=<id>` or `MISS`.
- **Exit code:** 0 on HIT, 1 on MISS, 2 on config/parse error.
- **Side effects:** Updates `last_accessed_at` for hit entries; increments stats.

### 7.2 `review-cache.sh record <diff-file> <reviewer-id> <verdict>`

- **Input:** Diff path, reviewer identifier (e.g., `kimi-rescue`), verdict (`PASS` or `FAIL`).
- **Output:** None (silent on success).
- **Exit code:** 0 always (FAIL verdicts are silently dropped).
- **Side effects:** Writes/updates hunk JSONs; increments `total_puts`; triggers LRU eviction if needed.

### 7.3 `review-cache.sh invalidate <file-path>`

- **Input:** Absolute or relative path to a source file.
- **Output (stdout):** `INVALIDATED <N> hunks for <path>`.
- **Exit code:** 0.
- **Side effects:** Deletes matching hunk entries.

### 7.4 `review-cache.sh stats`

- **Input:** None.
- **Output (stdout):** Human-readable table.
- **Exit code:** 0.
- **Side effects:** None.

## 8. Anti-Fixes (What NOT to do)

1. **Do NOT cache failing reviews.** A `FAIL` verdict means the code had issues. Reusing that verdict would ship un-fixed issues. The `record` command silently drops non-`PASS` verdicts.
2. **Do NOT use fuzzy matching.** No `difflib.SequenceMatcher`, no `git diff --patience`, no line-number normalization. Exact match only.
3. **Do NOT treat cache hits as unconditional LAND.** The review-cache skips the *reviewer-ant* gate. Queen-rerun gates (typecheck, test, lint) still execute at converge. This lever optimizes the expensive LLM-review step, not the cheap deterministic gates.
4. **Do NOT share cache across repos.** The cache root is global (`~/.claude/state/...`), but hunk hashes include the full file path. Two repos with `src/main.py` will not collide because the diff context (and ideally the repo path in the diff) differentiates them. If this becomes a problem, prefix the hash with a repo identifier in a future revision.
5. **Do NOT run this as a background daemon.** The cache is read-on-check, write-on-record, delete-on-invalidate. No GC thread, no watcher. Simplicity prevents leaks.

## 9. Future Work (Post-v2.22.1)

- **Repo-scoped cache:** Prefix `hunk_hash` with `repo_slug` to eliminate cross-repo collision risk.
- **Hierarchical invalidation:** Invalidate by directory prefix (e.g., `invalidate apps/api/` drops all hunks under that tree). Useful for large refactors.
- **Semantic hunk fingerprint:** Optional second-pass using tree-sitter node hashes. Would allow hits across line-offset changes at the cost of implementation complexity. Deferred until exact-match hit rate is measured as "too low."
- **Distributed cache:** For multi-host queens (v3.0), sync the cache via a shared blob store. TTL and LRU make this eventually-consistent safe.

## 10. Acceptance Criteria

- [ ] `check` returns HIT for a diff whose every hunk was previously recorded as PASS.
- [ ] `check` returns MISS if any hunk is missing, expired, or has no passing reviewer.
- [ ] `record` with `FAIL` writes nothing to the cache.
- [ ] `invalidate` removes all hunks for the target file.
- [ ] LRU evicts oldest entries when the 1,000-entry limit is exceeded.
- [ ] TTL expires entries older than 7 days.
- [ ] `mtime` mismatch causes a hit-to-miss promotion and deletes the stale entry.
- [ ] `stats` reports accurate hit rate, entry count, and estimated savings.
- [ ] No git commits are made by this draft.
